## Phase 1 (2026-04-29, commits 82c47df + 6488db1)

- **Sub-phase 1A discovered two pre-existing test-framework breaks** that fail on `main` HEAD before the flatten:
    1. `src/tests/integration/conftest.py` — `OracleAsyncConfig has no attribute pool_instance`. sqlspec ≥0.46 dropped the public `pool_instance` attribute. Fix: removed the assignment; `db.close_pool()` alone is sufficient.
    2. `test_vector_search_returns_typed_product_matches_with_price` — `ProductMatch.similarity_score` is NaN. Bootstrap seeds at least one product with a zero-norm embedding; Oracle's `VECTOR_DISTANCE(zero, x, COSINE)` is NaN; the WHERE filter `1 - VECTOR_DISTANCE(...) > 0.5` should reject NaN but at least one row slips through (Oracle's NaN-comparison semantics + `LOB_BY_VALUE` hint?). Worked around by tightening the conftest to truncate-and-load canonical fixtures via `FixtureLoader` (matches accelerator's pattern). This exposed a separate **FP-precision quirk**: cosine similarity of FP32-stored vectors against an FP64 query lands a few ULPs above 1.0 for self-matches → fixed the assertion bound to `[0, 1] ± 1e-6`.
- **Mypy "no attribute" trap** — `from app import config as app_config` doesn't tell mypy to follow the `config` submodule. Fix: `import app.config as app_config` (explicit submodule import). Caught when manage.py was rewritten in 1.5.
- **CLI restructure architectural test** — `test_coffee_help_does_not_construct_db_config` patches `app.config.db` constructor with a sentinel that raises and asserts `coffee --help` exits 0. This is the canary that proves `--help` no longer materializes the SQLSpec config — the whole reason for the rich_click rewrite. Without this test the regression slips back trivially.
- **Help-text parsing for subprocess CLI tests** — rich_click prints box-drawing chars + ANSI codes that defeat naive substring asserts. Added `_command_names_in_help()` helper that strips ANSI + box chars + parses the Commands panel structure. 13 subprocess tests in `test_cli_surface.py` use it.
- **xdist parallel safety for integration tests** — added `pytest_collection_modifyitems` applying `xdist_group(name="oracle_integration")` so all integration tests serialize on the shared Oracle pool. Without this, `-n 2 --dist=loadgroup` produced 6 concurrent-pool errors.
- **`@click.pass_context` type-ignore needed** in `_create_run_command()` — RichContext vs Context type mismatch; mirrors `dma/accelerator/src/py/dma/cli/commands/server.py:24-58` exactly (same `# type: ignore[arg-type]`).

## Phase 2 (2026-04-29, commits 83cb129 + 4a4373c)

- **Architectural pivot mid-phase: `domain/web/` peer-domain pattern** — original spec said put templates at `src/templates/` and the bundle output at `src/app/server/static/dist/`. After comparing with `dma/accelerator/src/py/dma/domain/web/static/web/` (their Vite bundle lives **inside** the Python web domain), pivoted to:
    - `src/resources/` — Vite INPUT (main.js, styles.css, public/) — sibling of `src/app/`, mirrors canonical jinja-htmx upstream
    - `src/app/domain/web/templates/` — Jinja templates (consumed by Litestar `TemplateConfig`)
    - `src/app/domain/web/static/dist/` — Vite OUTPUT (gitignored, `manage.py assets build` writes manifest.json + hashed bundles here)
    - Tailwind `@source` retargets to `../app/domain/web/templates` (relative to `src/resources/styles.css`)
- **litestar-vite has zero opinion on template location** — confirmed by inspecting `~/code/litestar/litestar-vite/litestar_vite/` source: no `template_dir`, no `templates_dir`, no `TemplateConfig` references. Templates are owned entirely by Litestar's `TemplateConfig(directory=Path(...))`. The plugin only registers Jinja globals (`vite()`, `vite_hmr()`, `vite_static()`, `vite_routes()`) into whatever Jinja engine is configured. `{{ vite('src/resources/main.js') }}` resolves via the input path string from `vite.config.ts` — must match exactly.
- **Sessions are already wired sqlspec-style** — `src/app/config.py:60-61` registers `StoreRegistry(stores={"sessions": OracleAsyncStore(config=db)})` + `ServerSideSessionConfig(store="sessions")`. Direct mirror of accelerator's `_StoreRegistry(stores={"sessions": AsyncpgStore(config=dbc)})` (Postgres). FlashPlugin needs only this middleware to function — no new infrastructure for cross-request notifications.
- **FlashPlugin is small + clean** — `litestar.plugins.flash.FlashPlugin(FlashConfig(template_config=...))`. Pushes via `flash(request, "msg", "category")`; renders via `{% for f in get_flashes() %}` Jinja global (pops from session — render-once semantics). Accelerator does NOT use FlashPlugin; we are the first.
- **`git rm -r src/js` does not delete gitignored content** — node_modules + .vite cache + .litestar.json are gitignored, so they remain on disk after `git rm`. Spec verification "ls src/js 2>&1 errors" requires a follow-up `rm -rf src/js`. Documented in this learning so future deletions know to combine both.
- **23 brand assets locked** — `git ls-files src/js/public/` returned exactly 23 (matches spec). `ls -la` showed 24 entries because of the `.` directory entry. The exact filename list is parametrized in `test_brand_assets_layout.py::test_each_brand_asset_present` as belt-and-braces against silent inventory drift.
- **HTMX wrapper template `partials/_chat_response.html.j2`** — composes `partials/message.html.j2` + OOB `partials/_metrics_badges.html.j2` + OOB `partials/_flash.html.j2` so a single `HTMXTemplate` returns three regions in one round-trip. Cleaner than a per-handler triple OOB concatenation; named consistently with the canonical jinja-htmx example's `book_card.html.j2` pattern.
- **`HTMXTemplate` composes other litestar-htmx response objects via kwargs** — `re_target=` → `Retarget`, `re_swap=` → `Reswap`, `push_url=` → `PushUrl`, `trigger_event=` + `after=` → `TriggerEvent`. The standalone classes are still useful for non-template responses (e.g., `Reswap("none")` after a `flash + side-effect` action).

## Phase 3 (2026-04-29, commit 12156d7)

- **`litestar-vite-plugin@0.22.x` ceiling is `.1`, not `.4`** — `npm view litestar-vite-plugin versions` shows the highest published 0.22.x is 0.22.1. Spec quoted "0.22.x" (range) so we pin floor 0.22.1. If a future bump to 0.23.x lands, revisit because the plugin's helpers API may have shifted.
- **Vite 8 + rolldown bundler key dance** — upstream `~/code/litestar/litestar-vite/examples/jinja-htmx/vite.config.ts:5` uses `const bundlerKey = Number(version.split(".")[0]) >= 8 ? "rolldownOptions" : "rollupOptions"` so the same config works against vite 7 (rollup) and vite 8 (rolldown). Carried forward verbatim. The `onwarn` hook suppresses HTMX's intentional eval usage so the build doesn't print noise.
- **`tsconfig.json` `baseUrl` triggers TS 7 deprecation** — upstream jinja-htmx tsconfig declares `baseUrl: "."` + `paths: {"@/*": [...]}`. With `typescript@5.9.3` this prints `Option 'baseUrl' is deprecated and will stop functioning in TypeScript 7.0`. Since we have one entry point and no path aliases this chapter, dropped both keys. If/when we need an alias, add `resolve.alias` in `vite.config.ts` instead — that keeps tsconfig forward-compatible.
- **Vite plugin's "no Litestar config detected" warning is informational** — when running `npm run build` from a pure-frontend context (no Python app importable from the plugin's auto-detection probe), litestar-vite-plugin prints a long "Continuing with defaults..." block and proceeds with the explicit options passed in `vite.config.ts`. Manifest still emits correctly. The warning will go away in Phase 4 once `ApplicationCore` + `ViteSettings` are flipped to `mode="template"`.
- **Build output: 647 kB raw (176 kB gzip) main bundle** — htmx (~50 kB) + Alpine (~50 kB) + ApexCharts (~500 kB) + helpers. ApexCharts is the dominant cost and is acceptable for a reference demo; documented in spec risks. If needed, defer ApexCharts to a dynamic import on the Explore page only — but Panel 4/5 use it at page load so the saving is small.
- **TDD red phase verification** — failing tests printed `FileNotFoundError: src/resources/styles.css` not the assertion message. That's the right failure mode (file genuinely missing), not a stale assertion against an existing-but-wrong file. Once Green hit, all 7 tests in the new files passed in 0.08s.
- **Brand assets auto-copy via Vite's `publicDir`** — the 23 brand assets ended up under `src/app/domain/web/static/dist/` (alongside `assets/`) because Vite's default `publicDir` is `<resourceDir>/public/` and it copies them as-is to the bundle output. No explicit copy step needed; `litestar-vite-plugin` honors this convention.

## Phase 4 (2026-04-29, commit d843d4e)

- **`TemplateConfig` lives at `litestar.template`, not `litestar.config.template`** — spec quoted the latter from older Litestar 2.x docs but the actual import is `from litestar.template import TemplateConfig` (verified via `find .venv -path '*litestar*' -name 'template*'`). Updating the spec for archaeology only — code uses the right path.
- **AsyncTestClient must be entered as a context manager or the lifespan never runs** — the original `client` fixture returned a bare `AsyncTestClient(app, ...)`. With Litestar's `dishka_lifespan` doing `await container.close()` only at lifespan exit, two tests in one process leak Dishka containers → leaked Oracle pool handles → second test fails with 500. Fix: rewrite both `client` and `htmx_client` as `async def` fixtures that `async with AsyncTestClient(...) as test_client: yield test_client`. This is **load-bearing** for any test that boots the real app twice in one process.
- **PageController belongs in `web/controllers/`, not `system/controllers/`** — spec line 464 said put `PageController` in `system/controllers/_pages.py` but that's a leftover from a pre-pivot draft (when the web peer-domain didn't exist). Moving it to `web/controllers/_pages.py` keeps controllers and templates co-located in the same domain — DomainPlugin auto-discovers it either way (controllers_by_domain log shows `'web': ['PageController']`).
- **ADK 2.0 rejects `app_name` strings with dashes** — `Runner(app_name="coffee-assistant", ...)` fails with `ValidationError: Invalid app name: must be a valid identifier`. Pre-existing bug in the chat path that surfaced when the first integration-style API test booted the chat handler. Renamed to `coffee_assistant` (3 sites in `services/adk.py`). Real fix in case ADK ever loosens the validation: leave as `coffee_assistant` since underscores work in both worlds.
- **Litestar `File.media_type` over `content_type`** — `File(...)` doesn't accept `content_type=`; the kwarg is `media_type=`. Without it, `.ico` files serve as `application/octet-stream` (the favicon test caught this). Setting `media_type="image/x-icon"` explicitly is the right call.
- **Dishka over-eagerness during validation paths** — Litestar resolves all handler parameters BEFORE running the handler body. So even on an "invalid input" path that would early-return, `Inject[ADKRunner]` triggers the full DI chain (which needs Oracle for `SQLSpecSessionService`). For unit-style API tests, monkeypatch ADKRunner.__init__ to a no-op AND ADKRunner.process_request to AsyncMock — both are needed because Dishka constructs the instance, not just calls the method. An autouse fixture in the test module covers all three test functions.
- **HTMXTemplate kwarg-composes other litestar-htmx response objects in practice** — `HTMXTemplate(re_target="#chat-error", re_swap="innerHTML")` correctly emits the `HX-Retarget` and `HX-Reswap` headers. Confirmed by `assert response.headers["HX-Retarget"] == "#chat-error"` in `test_htmx_validation_returns_chat_error`. No need to instantiate `Retarget(...)` / `Reswap(...)` separately when starting from a template response.
- **`flash(request, ...)` Jinja global is `get_flashes()`, not `get_flashed_messages()`** — Litestar's port of Flask-style flash uses `get_flashes()` per `litestar.plugins.flash`. Each call pops the messages from the session — render-once semantics. Caught when scaffolding `_flash.html.j2`.
- **`controllers = [...]` re-export in `__init__.py` triggers RUF067** — Ruff's `RUF067` rule says `__init__` should only contain docstrings + re-exports. Both `domain/system/controllers/__init__.py` and my new `domain/web/controllers/__init__.py` had `controllers: list[type[Controller]] = [...]` assignments. The list isn't actually consumed by `DomainPlugin` (it auto-discovers via `find_controllers_in_module()`). Dropped the assignment from web; left the system one alone (pre-existing).

## Phase 5 (2026-04-29, commit 3688bdd)

- **Engineering Conventions section added to spec.md mid-phase** — six rules surfaced from a live review during 5.1 implementation, applied to every Phase 5 endpoint and the pre-existing `vector_search_demo` it touched:
    1. Public APIs return typed msgspec Structs, never `dict[str, Any]`. Same for service methods that feed those handlers — a typed return beats handing the caller a dict to subscript.
    2. Direct schema imports (`from app.domain.system.schemas import MetricsSummary`) — the `from app.domain.X import schemas` namespace prefix is banned for new code.
    3. JSON I/O via `app.utils.serialization.from_json` / `to_json` (re-exports of `sqlspec.utils.serializers`, mirroring `dma/accelerator/src/py/dma/utils/serialization.py`). `json.loads` / `json.dumps` is banned for new code.
    4. `schema_type=` on every sqlspec `select` / `select_one_or_none` so rows map directly into Structs — no manual `int(row["col"] or 0)` casting at the Python level. Pair an internal `*Row` Struct with a public Struct when row shape and surface API differ (e.g. `CacheStatsRow` → `CacheStats`).
    5. `COALESCE(AVG(...), 0)` / `COALESCE(COUNT(*), 0)` on aggregates so rows arrive as concrete numbers — no `value or 0` Python null-handling.
    6. Inline single-use locals — `return await svc.x(self.validate_message(query))`, not `cleaned = self.validate_message(query); return await svc.x(cleaned)`. Bookkeeping locals that span branches (e.g. the `detailed_timings` dict in `vector_search_demo`) are fine.
- **`HTMXTemplate(push_url=...)` writes the `HX-Push-Url` header at construction time, no `.push_url` attribute** — looked for `response.push_url` in tests; doesn't exist. The kwarg goes straight into `response.headers`. Tests assert `response.headers["HX-Push-Url"] == "/explore?q=..."`.
- **`MetricsChart` / `TimeSeries` / `scatter_data` / `breakdown_data` were unused holdovers** — the old `MetricsChart` payload returned hardcoded empty arrays from a React-dashboard-era endpoint. Replaced with the Phase 5 `MetricsTimeSeries(labels, series: MetricsTimeSeriesPoints)` and added an internal `MetricsTimeSeriesRow` for `schema_type=` mapping. Same story for `MetricsSummary` — the four named per-card fields became `cards: list[MetricCard]` for `ls-for` consumption.
- **`test_named_sql_loading.py::test_no_inline_sql_strings_in_domain_services` is a real invariant** — my first stab at `explain_search_plan` had `f"EXPLAIN PLAN FOR {vector_sql}"` and a `"SELECT plan_table_output FROM TABLE(DBMS_XPLAN.DISPLAY())"` string in the service body. Test failed with `still contains inline SQL keywords: ['SELECT']`. Fixed by adding two named queries: `explain-plan-vector-search` (mirrors `vector-search-products` body, prefixed with `EXPLAIN PLAN FOR`) and `explain-plan-display` (the DBMS_XPLAN.DISPLAY pull). All SQL now lives in `src/app/db/sql/*.sql`.
- **404 body is empty under `raise_server_exceptions=False`** — `NotFoundException("classify-compare.json not found")` returns 404 with `content-length: 0` when the test client has `raise_server_exceptions=False`. The detail is consumed by the after-exception hook before serialization. Tests now assert only `status_code == 404`; the helpful "run the CLI" hint shows in real browser responses.
- **Module-scoped `CLASSIFY_COMPARE_PATH` lets tests redirect at a tmp file** — the controller reads the path from a module-level constant (`from app.domain.system.controllers import _explore; _explore.CLASSIFY_COMPARE_PATH = tmp_path / "classify-compare.json"`). Test fixture monkey-patches it per test. Cleaner than a class attribute or a settings-level path because it scopes the artifact to the controller that owns it.
- **Alpine `x-show` toggles over imperative DOM mutation** — first cut of the classify-compare-missing branch wrote the hint string into a host element imperatively. Rewrote the panel to render the hint inline with `x-data` boolean state + `x-show="missing"` + `x-cloak`. Same UX, no imperative DOM mutation, no XSS surface.
- **Ruff CPY001 enforces `Copyright YYYY Google LLC` header literally** — SPDX-style headers don't satisfy the rule. Phase 5 controllers / templates / tests all use the `# Copyright 2026 Google LLC` literal. Migration to SPDX style is tracked in epic `oracledb-vertexai-a0l`.

## Phase 6+7 (2026-04-29, commit 7dc4579)

- **`vite.config.ts` `publicDir` is project-root-relative, not resourceDir-relative** — Vite copies `<projectRoot>/<publicDir>` wholesale into `bundleDir` at build time. We had `resourceDir: "src/resources"` but no explicit `publicDir`, so Vite defaulted to `<repoRoot>/public/` (which doesn't exist) and silently shipped zero brand assets to the bundle dir. Surfaced when the chat-page logo 404'd in the Playwright smoke. Fix: add `publicDir: "src/resources/public"` to vite.config.ts. After the fix, `npm run build` copies all 27 brand assets (cymbal-coffee-cup.svg, favicon family, etc.) into `src/app/domain/web/static/dist/`. The litestar-vite plugin's `resourceDir` knob doesn't override Vite's publicDir resolution.
- **Granian `coffee run` ignores `VITE_DEV_MODE` env-var overrides when `.env` defines it** — `.env` had `VITE_DEV_MODE=True`. Setting `VITE_DEV_MODE=False` on the shell didn't switch dev-mode off because python-dotenv loads `.env` after the shell environment, overriding the export. To switch to production mode for the smoke, edited `.env` directly. Lesson: settings-loaded env vars take precedence over shell exports in this codebase.
- **Vite dev-mode auto-port-pick can race the `hot` file** — when `dev_mode=True` + `strictPort=5173` in `vite.config.ts`, the litestar-vite plugin still calls `find_unused_port()` and writes that port (e.g. 34259) to `.litestar.json`. The `hot` file gets `:5173` from strictPort. Templates render `vite_hmr()` from `.litestar.json` (port 34259), but Vite actually binds 5173 — connection refused on the auto-picked port. Workaround for the smoke: run in production mode. Fix proper is a litestar-vite-plugin upstream bug (track separately).
- **`make build` was Python-only before Phase 6.2** — only ran `uv build`, never the frontend bundle. So a fresh `make build` would emit an old wheel with stale assets. Phase 6.2 fixed: build now chains `manage.py assets build` then `uv build`, mirroring the install-time pairing.
- **`make lint` skipped frontend type-checking** — pyright + mypy + ruff covered Python, but `npx tsc --noEmit` only ran via the standalone `frontend-typecheck` target. Phase 6.2 chained `frontend-typecheck` after pyright so a single `make lint` invocation now covers both Python and TS.
- **The `uv run python manage.py assets install` indirection is the canonical install entry** — `manage.py assets install` calls `NodeExecutor.install()` which detects npm/pnpm/bun and runs the right install. Direct `npm install` works too but bypasses the executor selection. Both `make install` and the Dockerfile multi-stage builds go through `manage.py assets install` for consistency.
- **`coffee run --port` flag signature accepts both `--port=N` and `--port N`** — confirmed during smoke that `uv run coffee run --port 5099 --host 127.0.0.1` boots Granian on the requested port. The defaults in `.env` (`LITESTAR_PORT=5006`) are overridden by CLI flags.
- **Dockerfile `CMD` was `app run` not `coffee run`** — `app` was never registered as a console script in `pyproject.toml`; only `coffee` is. The pre-Ch-4 Dockerfile would have failed at container start. Phase 6.4 rewrite uses `CMD ["coffee", "run", ...]` to match the actual entry point.
- **Cumulative test inventory drift detection** — `test_explain_plan_shape` was specced under `src/tests/api/` but actually lives in `src/tests/unit/` (unit-style controller test, not real-Oracle integration). Spec corrected; the test invocation list now matches reality.
- **404 console errors on `/explore` are part of the contract** — Panel 5 fetches `/api/classify-compare` on mount; when the dist file is absent the endpoint returns 404 and Alpine flips `missing = true` to render the "run the CLI" hint via `x-show`. The browser DevTools console will show 2 ERR entries (panel renders twice during Alpine reactive bootstrap), which is **expected behavior** — _not_ a regression. Test invariants for the missing-file branch live in `test_classify_compare_endpoint.py`.
- **Test-locked toolchain invariants beat smoke-only verification** — Phase 6 added 14 new pyproject/Makefile/.gitignore/Dockerfile invariant tests to `test_repo_layout_invariants.py`. CI catches a re-introduction of `bun`, `litestar-htmx`, `coffee upgrade`, or `src/py` instantly. Without these, regressions slip in via well-meaning refactors.

## Phase 7.4 — Doc-touch (2026-04-29, commit 395fcf4)

- **`.agents/patterns.md` is for actionable rules, `.agents/knowledge/<flow_id>.md` is for verbatim learnings** — the convention here cleanly separates "what should I do?" from "what did we learn while doing it?". Patterns are concise and rule-shaped; knowledge entries preserve the per-phase backstory and key-files inventory. Both are committed; `index.md` is the registry that links knowledge entries by topic so future flows can grep by area (e.g. all `htmx`-tagged entries).
- **Ch 4 patterns folded in 6 engineering conventions on top of the 14 framework patterns** — the conventions (typed Structs, direct schema imports, `from_json`, `schema_type=`, `COALESCE`, inline locals) deserved their own section because they apply to *every new endpoint or service*, not just the HTMX/Vite surface. They're explicitly cross-cutting and originated mid-phase from user-driven review, so calling them out separately preserves the "these are non-negotiable for new code" framing.
- **Knowledge entry mirrors the spec's phase ordering** — verbatim learnings sectioned per phase (1 / 2 / 3 / 4 / 5 / 6+7) make it easy to drill into a specific decision without re-reading the whole spec. The Patterns Elevated list at the top is the executive summary; Key Files anchors the entry to actual code paths.
- **Topic-index entries multiply discoverability** — adding 10 topic tags (htmx, vite, alpine, tailwind, litestar, jinja, frontend, cli, oracle, explain-plan) means future flows looking for any of those concepts surface this knowledge entry. The cost is one line per topic; the benefit is order-of-magnitude better recall when seeding new specs.
