## Pre-implementation findings (planning phase, 2026-04-29)

- Accelerator's `lib/service.py` is a **pure re-export facade** ŌĆö there is no custom subclass. The codebase delegates everything to sqlspec's `SQLSpecAsyncService`. Mimic that exactly.
- Accelerator uses **5 providers** but the count is misleading: 3 of them (`LitestarPersistenceProvider`, `CliPersistenceProvider`, `WorkerPersistenceProvider`) are scope-specific copies of the same driver-providing logic. We only need the Litestar variant for now; CLI and Worker can be added if/when SAQ lands.
- The `current_price`/`price` mismatch at `services.py:47` has a band-aid mapping at line 119 that has been silently degrading vector-search response shape since the column was renamed. Reviewers missed it because the band-aid remapped onto the `distance` field ŌĆö type checker happy, semantics wrong.
- Filter dependencies are **purely additive** to the controller `dependencies` dict ŌĆö no risk of clobbering hand-rolled deps if the rewrite is methodical.

## Ch 2.1 ŌĆö Service base + facade (closed `[aad45ae]`, 2026-04-29)

- `lib/service.py` rewritten as a thin re-export of sqlspec primitives (`SQLSpecAsyncService`, filter types, `apply_filter`, `create_filter_dependencies`). Removed the local Generic[AsyncDriverT] wrapper.
- `sed -i 's/SQLSpecService/SQLSpecAsyncService/g'` across `src/py/app/` was clean ŌĆö no `__init__` overrides depended on the wrapper. 7 domain service classes now inherit directly from sqlspec.

## Ch 2.2 ŌĆö Named SQL extraction (closed `[c9a44a1]`, 2026-04-29)

- Refactored from 30 candidate keys down to **10 reusable named queries** by adopting `sqlspec.sql.{insert,update,delete}` AST builders for trivial CRUD. The general rule: only stash a query in `db/sql/*.sql` when it benefits from `.where(...)` chaining or has non-trivial SELECT shape.
- `db_manager.get_sql("...").where("predicate")` returns a chainable query object. Single-column-WHERE variants (`find_stores_by_city`, `_state`, `_id`) collapse to one base query + `.where(...)` calls ŌĆö saved 5 keys.
- Driver call sites must use **kwargs binds** (`select_one_or_none(query, id=product_id, schema_type=Product)`), not dict binds. The driver expects keyword arguments in sqlspec 0.46.
- Always pass `schema_type=` so the driver returns typed structs directly ŌĆö no manual `Schema(**row)` wrap needed.
- `SQLSpecAsyncService.paginate(query, *filters, schema_type=T)` replaces hand-rolled count+select pairs. Drops 2 separate named queries (`count-products`, `count-exemplars`).
- Hyphen vs underscore: sqlspec's registry stores keys with underscore but `get_sql("hyphen-name")` works because the loader normalizes. Test regex must require **whitespace lookahead** (`(?=[ \t\n])`) after `SELECT/UPDATE/...` to avoid false positives on hyphenated query keys like `"update-product-embedding"`.
- CLI `force=True` branch was hiding inline SQL (`SELECT id, name, description, embedding FROM product`). Eliminated by extending `ProductService.get_products_for_embedding(force=)` to mirror `ExemplarService.get_exemplars_without_embeddings`.

## Ch 2.3 ŌĆö 3-provider Dishka collapse (closed `[e49131f]`, 2026-04-29)

- `IntegrationsProvider` holds **only** the truly APP-scoped externals: `genai.Client`, `OracleAsyncADKStore`, `SQLSpecSessionService`, `ADKRunner`. `VertexAIService` and `OracleVectorSearchService` were proposed APP-scoped in the spec but stay **REQUEST-scoped** in `DomainServiceProvider` because `VertexAIService` captures a REQUEST-scoped `CacheService` reference at construction time ŌĆö promoting it to APP would pin a stale per-request reference. Spec text refined accordingly.
- `provide_intent_classifier` slot deferred: YAGNI wins until Ch 3 actually lands `FlashLiteIntentClassifier`. Reserving an empty provider method now would be dead code.
- `ADKRunner(session_service)` signature unchanged. The spec mentioned `(session_service, classifier, persona_manager)` but `PersonaManager` is a static class today (`PersonaManager.get_system_prompt(...)` is called as a class method), not an injected instance. No Ch 3 wiring change is required here.
- Dishka `AsyncContainer` exposes no public provider list; introspection requires walking `container.registry.child_registry` chain (one registry per scope). For test stability, the implementation also exposes a `PROVIDERS: tuple[type[Provider], ...]` module-level constant ŌĆö tests assert against this rather than a private API.
- Lint trap: this module **must not** carry `from __future__ import annotations`. Dishka uses `get_type_hints()` at provider-init time, which under PEP 563 sees string forward refs and fails. Encoded as a guard test (`test_ioc_module_does_not_use_future_annotations`).
- Existing `test_chat_di.py` monkeypatched `ChatServiceProvider.get_*` methods. After the collapse those methods live on `IntegrationsProvider`. Test rewritten one-to-one ŌĆö same DI semantics, different host class.

## CLI flatten + accelerator manage.py (parallel work, 2026-04-29)

- Application CLI command renamed from `app` to `coffee` (`[project.scripts] coffee = "app.__main__:run_cli"`). All Makefile + README + CLAUDE.md references migrated.
- `[tool.sqlspec] config = "app.config.db"` added to pyproject so the standalone `sqlspec` CLI auto-discovers the config (matches accelerator). The custom `manage.py database` group still pre-injects `ctx.obj["configs"]` directly ŌĆö the pyproject entry is a fallback for direct `sqlspec` invocations.
- `manage.py` now mirrors dma/accelerator: 6 top-level groups (`init`, `install`, `doctor`, `infra`, `database`, `assets`). Container ops are flat (`infra start/stop/restart/status/logs/wipe`) ŌĆö the `-local-container` suffix die. `wallet` and `connect` bundle under `database` so DB-related concerns cluster.
- Application CLI flattens too: `coffee upgrade/downgrade/load-fixtures/export-fixtures/bulk-embed/clear-cache/model-info` are top-level. The sqlspec auto-mounted `db | database` group is suppressed via a `SQLSpecPlugin` subclass whose `on_cli_init` is a no-op (the upstream plugin auto-mounts; our `ApplicationCore` plugin runs first so `cli.commands.pop("db")` doesn't take ŌĆö subclass override is the right hammer).
- `async_inject` decorator ported from accelerator to `src/py/app/cli/utils.py`. Each coffee command now declares its dependencies as type-annotated kwargs (`product_service: ProductService`, etc.); the decorator builds a fresh REQUEST-scoped Dishka container, resolves the deps, sets `worker_container_var` and `request_container_var`, and runs the coroutine via `sqlspec.utils.sync_tools.run_`. Eliminates the manual `db_manager.provide_session(db) as session: ProductService(session); CacheService(session); ...` boilerplate per command.
- Trap: with `from __future__ import annotations`, `get_type_hints()` evaluates string annotations against module globals ŌĆö so the service classes used as deps **must be imported at runtime** (not under `TYPE_CHECKING`). Marked with `# noqa: TC001` so ruff doesn't push them back into the type-checking block.

## Ch 2.4 ŌĆö Filter dependencies + paginated handlers (closed `[4167a45]`, 2026-04-29)

- `create_filter_dependencies({...})` emits the dep keys `{filters, limit_offset_filter, search_filter, id_filter, order_by_filter, created_filter}` ŌĆö the *configuration* keys (`pagination_type`, `sort_field`, `id_field`, `search`, `created_at`) are **input shape**, not output shape. Tests assert against the actual emitted keys. No rename shims (per user direction): the tests adapt to sqlspec's vocabulary, not the other way around.
- `B008` is now globally ignored (`pyproject.toml`). The Litestar idiom `filters: list[FilterTypes] = Dependency(skip_validation=True)` is a function call in a default-value slot ŌĆö flake8-bugbear's general rule against this is wrong for Litestar's design; accelerator ignores B008 for the same reason.
- New schema gotcha (cousin of the Dishka PEP 563 trap): handler-visible msgspec structs with `from __future__ import annotations` AND `if TYPE_CHECKING: from datetime import datetime` break Litestar's OpenAPI generator at boot. Litestar calls `get_type_hints()` to materialize response models ŌĆö string forward refs fail because `datetime` isn't in module globals. Fix: import `datetime` at runtime with `# noqa: TC003`. This bit `Product`/`Store` (already shipped, just newly exposed via handlers) and the new `IntentExemplar`. Worth promoting to a top-level patterns.md note in Ch 2.7.
- `list-products` was added as a new named query specifically for list endpoints. The existing `get-product` projects `embedding` (3072 dims) ŌĆö heavy for list responses. `list-products` projects everything **except** embedding. Same logic applied to widening `list-exemplars`. Pattern: keep one-row queries fat, list queries lean.
- `paginate(query, *filters, schema_type=T)` does the count+select+ordering+offset internally ŌĆö services collapse to a one-liner. Tests assert the return-type annotation is `OffsetPagination[T]` via `typing.get_origin`/`get_args` so the contract is type-checked at unit level (no DB needed).
- Phase 4.4 was a no-op confirmation: a thorough `grep -rn -E "(query_param|request\.url\.query|Parameter.*limit|Parameter.*offset|skip\s*=|page_size)" src/py/app/domain/ src/py/app/server/` returned zero hits. The codebase had no hand-rolled pagination to remove.
- Domain layout: kept `controllers.py` flat in this chapter even though the spec sample shows `controllers/_product.py`. The DomainPlugin auto-discovers both flat files and packages, so adding ProductController/StoreController/ExemplarController to existing flat files keeps the diff scoped. Phase 5 normalizes the layout to packages ŌĆö that's the right place for the structural churn.

## Ch 2.5 ŌĆö Domain layout normalization (closed `[fe931ba]`, 2026-04-29)

- Spec's Phase 5.1 sample (inline `pkgutil.iter_modules` in `ApplicationCore.on_app_init`) does NOT match dma's actual `accelerator/src/py/dma/server/core.py`, which delegates to a `DomainPlugin` (1:1 mirror of `app/utils/domains.py`). Inlining would *duplicate* the plugin's discovery layer. Refinement: kept `ApplicationCore` as-is; satisfied the spec's structural intent through the explicit `controllers: list[type[Controller]]` contract added per package in 5.2.
- Each `controllers/__init__.py` now exports BOTH `__all__` (class names ŌĆö for re-export discoverability) AND `controllers: list[type[Controller]]` (explicit contract ŌĆö survives any future move to inline pkgutil discovery). The DomainPlugin still uses class introspection so the redundancy is intentional: `__all__` is for human/IDE consumers, `controllers` is for ApplicationCore's contract.
- 8 expected routes confirmed via boot smoke (`app.openapi_schema.to_schema()`): `/api/{chat, exemplars, metrics/charts, metrics/summary, products, stores, vector-demo}` + `/metrics`. The new package layout works transparently with the existing `DomainPlugin.discover_domain_controllers` walk because `pkgutil.walk_packages` recurses into `controllers/` packages just as it did flat `controllers.py` modules.
- Test strategy mirrors Ch 2.3's `PROVIDERS` constant approach: introspect at module level (`getattr(module, "controllers")`, `getattr(module, "__all__")`) instead of asserting on private DomainPlugin internals. 13 tests (3 packages ├Ś 3 contract assertions + 1 negative-existence test for the flat file + 1 cross-domain discovery integration check).
- Tests must use `Path(module.__file__).name == "__init__.py"` to assert the package layout ŌĆö checking only `hasattr(__path__)` is necessary but not sufficient because Python's namespace packages also expose `__path__` without an `__init__.py`. The `__init__.py` check is the defensive form.
- `clear_discovery_cache()` is mandatory before the cross-domain integration test in this file ŌĆö the `_DiscoveryCache` in `app/utils/domains.py` survives across pytest runs; without the reset, a stale cache from a prior test in the suite (running with the OLD flat layout in development) would mask discovery breakage.
- One file per Controller class (mirror dma): chat split is trivial (1 class ŌåÆ 1 file `_chat.py`); products split into `_products.py` (Product/Store, both list-endpoint controllers) + `_vector.py` (one-off vector demo). Grouping by *concern* ("two list endpoints share a file because they share the create_filter_dependencies idiom") feels more readable than the strict "one class per file" ŌĆö accelerator does this too (e.g., `_mfa.py` holds both MFAController and MFAChallengeController).

## Ch 2.6 ŌĆö `current_price` bug fix (closed `[6cc3ca2]`, 2026-04-29)

- The SQL half (`db/sql/products.sql`) and the `Product` DTO half were already absorbed during Ch 2.2's named-SQL extraction. `grep -rn current_price src/py/app/` returned zero hits coming into 2.6. The actual remaining work was Python-side: deleting the `r["distance"] = 1 - r["similarity_score"]` band-aid in `OracleVectorSearchService.similarity_search` and the inverse `1 - r["distance"]` arithmetic in the vector-demo controller.
- **Naming convention (user direction this session): no "DTO" suffixes. msgspec Structs are named after the entity** (`Product`) **with intent suffixes for inputs only** (`ProductCreate`, `ProductUpdate`). The vector-search projection became `ProductMatch` (a Product-shaped row + match metadata) ŌĆö *not* `VectorSearchResult` as the spec text originally suggested. This generalizes: any future "X-with-extra" projection should use `<Entity><Qualifier>` (entity-first) rather than a verb-phrase or "Result"/"DTO" tail.
- **Move math to SQL where possible (user direction this session)**: the named query already produces `1 - VECTOR_DISTANCE(...) AS similarity_score` (0-1 range). Anything callers want to compute should be derived from that single projection, not re-derived via a band-aid round-trip. The remaining `* 100` in the controller is *display formatting* (round to 1 decimal for the UI) and is fine in Python ŌĆö the test of "math vs formatting" is whether the value carries semantic meaning beyond presentation.
- The unused `VectorDemoResult` Struct was dead code (defined + exported but never referenced anywhere). Removed during this chapter under the "no dead code" mandate. Lesson: when a chapter introduces a new typed Struct (here, `ProductMatch`), also audit *exports* in the same `schemas/__init__.py` for stale companions.
- **`schema_type=` makes the band-aid impossible to add back**: with `search_by_vector(...) -> list[ProductMatch]` (msgspec Struct), trying to mutate `r["distance"] = ...` would raise `TypeError: 'ProductMatch' object does not support item assignment`. The type system enforces the contract ŌĆö tests are belt-and-suspenders, not the only defense.
- **Test framework gap, addressed**: prior to this chapter, vector search had **zero unit or integration coverage** despite being one of the demo's headline features. Added 4 unit tests (no DB ŌĆö static SQL assertions, AsyncMock service test, controller `.fn` shape test) and 2 integration tests (real Oracle via the existing `driver` fixture; seeds a known embedding via UPDATE on the bootstrap-seeded product, asserts self-query yields `similarity_score Ōēł 1.0`). The unit tests run on every dev box; the integration tests gate CI but degrade gracefully (driver fixture errors out at connect time when no Oracle is reachable).
- Integration test seed pattern: the conftest's `driver` fixture bootstraps the `product` table + 1 seed row (no embedding). To exercise vector search, do a direct `UPDATE product SET embedding = :embedding WHERE sku = 'SEED-SKU-001'` with a deterministic 3072-dim FLOAT32 vector (used `[0.5] * 3072` ŌĆö equal-magnitude is well-defined under cosine; all-zero would trigger Oracle's "VECTOR_DISTANCE: norm undefined" error).
- `Path(__file__).resolve().parents[2]` from `tests/unit/test_*.py` lands at `src/py/`, **not** the repo root. Static-text tests that read `app/db/sql/products.sql` should use that path, mirroring `test_named_sql_loading.py`.
